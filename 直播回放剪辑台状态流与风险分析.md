# 直播回放剪辑台状态流与风险分析

## 一、整体状态流梳理

### 1.1 状态流全景图

```
┌─────────────────┐     ┌─────────────────┐     ┌─────────────────┐
│   台词切片 Tab   │────▶│   弹幕重排 Tab   │────▶│   音效试听 Tab   │
│  (slicing)      │     │  (danmaku)      │     │  (sfx)          │
└─────────────────┘     └─────────────────┘     └─────────────────┘
          │                        │                        │
          ▼                        ▼                        ▼
┌─────────────────┐     ┌─────────────────┐     ┌─────────────────┐
│  segments 数组   │────▶│   创建版本      │────▶│  版本管理/导出   │
│  (工作区)        │     │  ClipVersion    │     │  (export)       │
└─────────────────┘     └─────────────────┘     └─────────────────┘
                                 │
                                 ▼
                        ┌─────────────────┐
                        │  版本对比 Tab   │
                        │  (compare)      │
                        └─────────────────┘
```

### 1.2 核心状态结构

#### ClipEditorState 主状态
| 字段 | 类型 | 说明 |
|------|------|------|
| `activeTab` | `ClipTab` | 当前激活的标签页 |
| `selectedNodeId` | `string \| null` | 当前选中的源节点 |
| `segments` | `ClipSegment[]` | **工作区片段数组**（实时编辑） |
| `selectedSegmentId` | `string \| null` | 当前选中的片段 |
| `versions` | `ClipVersion[]` | **版本列表**（快照存储） |
| `selectedVersionIds` | `string[]` | 用于对比的选中版本 |
| `currentVersionId` | `string \| null` | 当前编辑的版本 |

#### ClipSegment 片段结构
| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | `string` | 片段唯一标识 |
| `name` | `string` | 片段名称 |
| `nodeId` | `string` | 关联的源节点ID |
| `startDialogueIndex` | `number` | 起始对白索引 |
| `endDialogueIndex` | `number` | 结束对白索引 |

#### ClipVersion 版本结构
| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | `string` | 版本唯一标识 |
| `name` | `string` | 版本名称 |
| `segments` | `ClipSegment[]` | **片段快照**（创建时深拷贝） |
| `danmakuOrder` | `Record<string, string[]>` | 各片段的弹幕排序 |
| `sfxOverrides` | `Record<string, AudioTrigger[]>` | 音效覆盖 |

### 1.3 关键状态流转路径

**路径1：片段创建**
```
1. selectClipNode(nodeId)
   → 设置 selectedNodeId
   → 重置 selectedSegmentId = null

2. addSegment(nodeId, startIdx, endIdx, name?)
   → 生成新 ClipSegment
   → 追加到 state.segments 数组
   → 设置 selectedSegmentId = 新片段ID
```

**路径2：版本创建**
```
createVersion(name)
   → 读取当前 state.segments
   → JSON.parse(JSON.stringify(segments)) 深拷贝
   → 创建新 ClipVersion
   → 追加到 state.versions
   → 设置 currentVersionId = 新版本ID
```

**路径3：弹幕重排**
```
moveDanmaku(segmentId, danmakuId, newIndex)
   ├─ 检查 currentVersionId（无版本则静默失败）
   ├─ 从 version.danmakuOrder 获取当前顺序
   │  └─ 如无记录，从 node.danmakus 动态构建
   ├─ 数组 splice 操作重排
   └─ 更新 version.danmakuOrder[segmentId]
```

**路径4：版本导出**
```
exportClipVersion(versionId)
   ├─ 获取版本的 segments 快照
   ├─ 遍历 segments，从 allNodes 重新拉取原始数据
   │  ├─ 切片 dialogues
   │  └─ 过滤并重定位 danmakus 的 dialogueIndex
   ├─ 合并多个 segment 到同一 node
   └─ 序列化为 JSON
```

## 二、拖拽重排的隐性风险

### 2.1 风险概览

| 风险ID | 风险类型 | 严重程度 | 影响范围 |
|--------|----------|----------|----------|
| DR-01 | 静默失败 | ⚠️ 中 | 所有拖拽操作 |
| DR-02 | 数据不同步 | ⚠️ 高 | 弹幕显示逻辑 |
| DR-03 | 边界检查缺失 | ⚠️ 中 | 拖拽索引 |
| DR-04 | 并发状态不一致 | ⚠️ 低 | 快速连续拖拽 |
| DR-05 | 无撤销机制 | ⚠️ 中 | 用户误操作 |

---

### 2.2 详细风险分析

#### DR-01: 无版本时拖拽操作静默失败

**问题代码位置**：`clipEditorStore.ts:155-158`
```typescript
export function moveDanmaku(segmentId: string, danmakuId: string, newIndex: number) {
  clipEditorState.update(s => {
    const version = s.versions.find(v => v.id === s.currentVersionId);
    if (!version) return s;  // 直接返回，无任何提示
```

**风险描述**：
- UI 层（`ClipEditor.svelte:161-181`）允许用户开始拖拽操作
- 但如果用户尚未创建版本，`moveDanmaku` 会静默失败（直接 `return s`）
- 用户看到拖拽视觉反馈，但实际数据未改变
- 没有错误提示，用户无法感知操作失败

**代码溯源**：
- 拖拽开始：`ClipEditor.svelte:161-163` → `handleDragStart`
- 拖拽放置：`ClipEditor.svelte:170-176` → `handleDrop` → `moveDanmaku`
- 版本检查缺失：UI 未在 `handleDragStart` 时检查 `version` 是否存在

**改进建议**：
1. 在 `handleDragStart` 中检查 `version`，无版本时阻止拖拽
2. 在 `moveDanmaku` 失败时返回错误码或抛出事件
3. UI 层添加版本缺失的视觉提示

#### DR-02: 弹幕顺序双源数据不同步

**问题代码位置**：`ClipEditor.svelte:191-206`

```typescript
function getOrderedDanmakus(): Danmaku[] {
  if (!segment || !version) return danmakus;
  const order = version.danmakuOrder[segment.id];
  if (!order || order.length === 0) return danmakus;
  
  const dmMap = new Map(danmakus.map(d => [d.id, d]));
  const ordered: Danmaku[] = [];
  for (const id of order) {
    const dm = dmMap.get(id);
    if (dm) ordered.push(dm);
  }
  // 未在 order 中的弹幕追加到末尾
  for (const dm of danmakus) {
    if (!order.includes(dm.id)) ordered.push(dm);
  }
  return ordered;
}
```

**风险描述**：
- 存在两个数据源：
  1. `version.danmakuOrder[segmentId]` → 存储的排序
  2. `node.danmakus` → 原始弹幕列表
- `getOrderedDanmakus()` 运行时合并，但**未持久化合并结果**
- 如果源节点新增弹幕，新弹幕会被追加到末尾，但不会自动加入 `danmakuOrder`
- 再次调用 `moveDanmaku` 时，如果 `danmakuOrder` 中没有该弹幕ID，会从 `node.danmakus` 重建顺序（`clipEditorStore.ts:164-178`），这会**丢失之前的自定义排序**

**关键时序风险**：
```
用户操作序列：
1. 创建版本 V1，danmakuOrder = { seg1: [d1, d2, d3] }
2. 在编辑器中修改源节点，新增弹幕 d4
3. 返回剪辑台拖拽 d3 到首位
4. moveDanmaku 发现 d3 不在 order 中（因为 order 是 V1 创建时的快照）
5. 从 node.danmakus 重建顺序 → [d1, d2, d3, d4]
6. 用户之前的排序 [d3, d1, d2] 丢失！
```

**改进建议**：
1. 版本创建时同时存储 `danmakuOrder` 的初始快照
2. `moveDanmaku` 不动态重建顺序，而是直接使用版本中存储的顺序
3. 检测到源节点弹幕变化时，提示用户更新版本

---

#### DR-03: 拖拽索引边界检查缺失

**问题代码位置**：`clipEditorStore.ts:181-183`

```typescript
const newOrder = [...currentOrder];
newOrder.splice(oldIndex, 1);
newOrder.splice(newIndex, 0, danmakuId);
```

**风险描述**：
- `newIndex` 参数未做边界验证
- 如果 `newIndex < 0`，`splice` 会从数组末尾开始计数
- 如果 `newIndex >= newOrder.length`，`splice` 会直接追加到末尾
- 快速拖拽或鼠标移出列表区域时可能触发越界索引
- 虽然 `splice` 不会崩溃，但行为可能不符合用户预期

**改进建议**：
```typescript
// 添加边界检查
const safeIndex = Math.max(0, Math.min(newIndex, currentOrder.length - 1));
newOrder.splice(safeIndex, 0, danmakuId);
```

---

#### DR-04: 并发拖拽状态不一致

**问题代码位置**：`ClipEditor.svelte:63-64, 161-181`

```typescript
let draggedDanmakuId: string | null = null;
let dragOverIndex: number | null = null;
```

**风险描述**：
- 拖拽状态是组件局部变量，未与 store 同步
- 快速连续拖拽时：
  1. 第一次拖拽的 `handleDrop` 异步执行
  2. 第二次拖拽已经开始，`draggedDanmakuId` 被覆盖
  3. 第一次 `handleDrop` 可能使用了错误的 `draggedDanmakuId`
- 虽然概率较低，但在低性能设备上可能发生

**改进建议**：
1. 添加拖拽锁机制，防止并发拖拽
2. 在 `handleDrop` 中验证 `draggedDanmakuId` 是否匹配

---

#### DR-05: 拖拽操作无撤销机制

**风险描述**：
- `moveDanmaku` 直接修改版本数据，无历史记录
- 用户误操作后只能点击"重置顺序"按钮（`ClipEditor.svelte:210-215`）
- "重置顺序"会丢失所有自定义排序，无法部分撤销
- 没有操作栈或历史记录

**改进建议**：
1. 实现简单的操作历史栈（最多保留20步）
2. 提供 Ctrl+Z 快捷键支持
3. "重置顺序"前提示用户确认

## 三、版本切换的隐性风险

### 3.1 风险概览

| 风险ID | 风险类型 | 严重程度 | 影响范围 |
|--------|----------|----------|----------|
| VS-01 | 工作区与版本不同步 | ⚠️ 高 | 所有版本操作 |
| VS-02 | 深拷贝数据丢失 | ⚠️ 中 | 版本创建 |
| VS-03 | 版本切换不恢复数据 | ⚠️ 高 | 多版本编辑 |
| VS-04 | 版本删除引用悬空 | ⚠️ 中 | 对比视图 |
| VS-05 | 导入数据验证不足 | ⚠️ 高 | 版本导入 |
| VS-06 | 导出数据与版本不一致 | ⚠️ 中 | 版本导出 |

---

### 3.2 详细风险分析

#### VS-01: 工作区 segments 与版本 segments 双轨并行

**问题代码位置**：`clipEditorStore.ts:199-220`

```typescript
export function createVersion(name: string): string {
  const newId = `ver_${Date.now()}`;
  const $state = get(clipEditorState);
  
  const newVersion: ClipVersion = {
    id: newId,
    name,
    segments: JSON.parse(JSON.stringify($state.segments)),  // 快照
    danmakuOrder: {},
    sfxOverrides: {},
    createdAt: Date.now(),
    updatedAt: Date.now()
  };
  
  clipEditorState.update(s => ({
    ...s,
    versions: [...s.versions, newVersion],
    currentVersionId: newId
  }));
```

**风险描述**：
- 系统中存在**两套独立的 segments 数据**：
  1. `state.segments` → 工作区实时数据，用户在"台词切片"Tab 中编辑
  2. `version.segments` → 版本快照，创建时深拷贝
- 创建版本后，用户继续在"台词切片"Tab 中修改片段：
  - 修改的是 `state.segments`
  - **不会自动同步**到 `currentVersion.segments`
  - 但弹幕编辑 `moveDanmaku` 是写入 `currentVersion.danmakuOrder`
- 用户容易混淆：以为在"弹幕重排"Tab 的修改是基于当前看到的片段，但实际弹幕排序是绑定到版本创建时的片段快照

**用户认知误区**：
```
用户预期：
  调整片段范围 → 切换到弹幕Tab → 拖拽排序
  → 排序基于调整后的片段

实际行为：
  调整片段范围 → 修改的是 state.segments
  切换到弹幕Tab → 使用的是 currentVersion.segments（创建时的快照）
  → 两者可能不一致！
```

**改进建议**：
1. 创建版本后，锁定"台词切片"Tab 的编辑功能
2. 或提供"更新当前版本"按钮，将工作区同步到版本
3. UI 清晰标示当前编辑的是"工作区"还是"版本"

---

#### VS-02: JSON 深拷贝的潜在数据丢失

**问题代码位置**：`clipEditorStore.ts:206`

```typescript
segments: JSON.parse(JSON.stringify($state.segments)),
```

**风险描述**：
- `JSON.parse(JSON.stringify())` 是简单深拷贝，但有局限性：
  - 丢失 `undefined` 值
  - 丢失函数
  - 丢失 `Symbol`
  - 无法处理循环引用
  - `Date` 对象会变成字符串
- 当前 `ClipSegment` 都是简单类型，暂时安全
- 但未来如果扩展 `ClipSegment`，添加复杂类型字段（如 `Date`、函数），会静默丢失数据

**改进建议**：
1. 使用结构化克隆 `structuredClone()`（现代浏览器支持）
2. 或实现类型安全的深拷贝函数
3. 添加单元测试验证拷贝完整性

---

#### VS-03: 版本切换不恢复工作区状态

**问题代码位置**：`clipEditorStore.ts:254-256`

```typescript
export function setCurrentVersion(versionId: string | null) {
  clipEditorState.update(s => ({ ...s, currentVersionId: versionId }));
}
```

**风险描述**：
- `setCurrentVersion` 只更新 `currentVersionId` 字段
- **不会**将版本中的 `segments` 恢复到工作区 `state.segments`
- **不会**将版本中的 `danmakuOrder` 同步到任何显示状态
- 导致：
  1. 用户切换版本后，"台词切片"Tab 显示的还是旧的工作区片段
  2. "弹幕重排"Tab 显示的是新版本的弹幕排序
  3. 两个 Tab 显示的数据来源不一致，造成认知混乱

**时序示例**：
```
1. 创建版本 V1（segments: [A, B]）
2. 在工作区新增片段 C → segments: [A, B, C]
3. 切换到其他版本 V2 → currentVersionId = V2
4. 切回 V1 → currentVersionId = V1
5. "台词切片"Tab 仍显示 [A, B, C]（工作区）
6. "弹幕重排"Tab 显示 V1 的排序（基于 [A, B]）
→ 两个 Tab 数据不一致！
```

**改进建议**：
1. `setCurrentVersion` 时，询问用户是否将版本数据恢复到工作区
2. 或提供明确的"恢复到工作区"按钮
3. UI 清楚标示当前工作区与所选版本的差异

---

#### VS-04: 版本删除后的引用悬空

**问题代码位置**：`clipEditorStore.ts:231-243`

```typescript
export function deleteVersion(versionId: string) {
  clipEditorState.update(s => {
    const newVersions = s.versions.filter(v => v.id !== versionId);
    return {
      ...s,
      versions: newVersions,
      currentVersionId: s.currentVersionId === versionId 
        ? (newVersions[newVersions.length - 1]?.id || null)
        : s.currentVersionId,
      selectedVersionIds: s.selectedVersionIds.filter(id => id !== versionId)
    };
  });
}
```

**风险描述**：
- 删除版本时正确处理了 `currentVersionId` 和 `selectedVersionIds`
- 但存在时序问题：
  1. 用户在"版本对比"Tab，选中了版本 V1 和 V2
  2. `diffResult` 已计算完成并显示
  3. 用户删除 V1
  4. `selectedVersionIds` 被更新，但 `diffResult` 是派生状态
  5. Svelte 的响应式更新 `$: diffResult = ...` 会重新计算
  6. 但如果在对比模态框中，可能出现短暂的不一致

**改进建议**：
1. 删除版本前检查是否在对比中，提示用户
2. 删除后自动关闭对比视图

#### VS-05: 导入版本数据验证不足

**问题代码位置**：`clipEditorStore.ts:321-347`

```typescript
export function importClipVersion(jsonData: string): boolean {
  try {
    const data = JSON.parse(jsonData);
    if (!data.version || !data.segments || !data.nodes) return false;
    
    const newId = `ver_${Date.now()}`;
    const newVersion: ClipVersion = {
      id: newId,
      name: `${data.version} (导入)`,
      segments: data.segments,
      danmakuOrder: data.danmakuOrder || {},
      sfxOverrides: data.sfxOverrides || {},
      createdAt: Date.now(),
      updatedAt: Date.now()
    };
```

**风险描述**：
- 只做了最基本的字段存在性检查
- **未验证**：
  1. `data.segments` 中每个 segment 的结构完整性
  2. `segment.nodeId` 是否存在于当前 `allNodes` 中
  3. `danmakuOrder` 中的弹幕ID是否有效
  4. `segments` 数组是否为空
  5. 数据版本兼容性
- 导入损坏或恶意构造的数据可能导致后续运行时错误

**改进建议**：
1. 使用 Zod 或类似的验证库做运行时类型验证
2. 验证 `segments` 中引用的 `nodeId` 确实存在
3. 版本号检查，不兼容版本给出提示

#### VS-06: 导出数据与版本快照不一致

**问题代码位置**：`clipEditorStore.ts:273-306`

```typescript
export function exportClipVersion(versionId: string): string {
  const version = get(clipEditorState).versions.find(v => v.id === versionId);
  if (!version) return '';
  
  const nodesMap = new Map<string, StoryNode>();
  const $allNodes = get(allNodes);
  
  for (const segment of version.segments) {
    const origNode = $allNodes.find(n => n.id === segment.nodeId);
    // 从当前 allNodes 重新拉取数据，不是使用版本中存储的数据
```

**风险描述**：
- 导出时**不是**直接序列化版本中存储的数据
- 而是重新从 `allNodes`（当前编辑器中的节点数据）拉取原始数据
- 如果版本创建后，源节点被修改（对白、弹幕被编辑）：
  - 导出的数据会是**修改后**的最新数据
  - 而不是版本创建时的快照数据
- 违反了"版本是快照"的用户预期

**用户预期 vs 实际**：
```
用户预期：
  创建版本 V1（基于节点 N 的状态 S1）
  后续修改节点 N 到状态 S2
  导出 V1 → 得到 S1 的数据

实际行为：
  导出 V1 → 得到 S2 的最新数据
  → 版本的快照意义被破坏！
```

**改进建议**：
1. 版本创建时同时保存相关节点数据的快照
2. 或导出时明确提示"将导出最新节点数据"
3. 提供两种导出模式：快照模式 / 最新数据模式

## 四、综合风险等级评估

### 4.1 高优先级风险（建议立即修复）

1. **VS-01 工作区与版本不同步**
   - 影响：用户认知混乱，数据错误
   - 建议：添加版本更新机制，UI 明确标示

2. **VS-03 版本切换不恢复数据**
   - 影响：多版本编辑时数据不一致
   - 建议：版本切换时提供数据恢复选项

3. **DR-02 弹幕顺序双源不同步**
   - 影响：自定义排序意外丢失
   - 建议：弹幕排序以版本存储为准，不动态重建

4. **VS-05 导入数据验证不足**

## 五、架构设计建议

### 5.1 状态模型重构建议

**当前问题**：双轨数据模型（工作区 + 版本快照）容易混淆

**建议方案**：采用单一数据源模型
```
┌─────────────────────────────────────┐
│         版本（ClipVersion）         │  ← 唯一真相来源
│  ├─ segments: ClipSegment[]         │
│  ├─ danmakuOrder                    │
│  └─ sfxOverrides                    │
└─────────────────────────────────────┘
               ▲
               │ 编辑操作直接作用于版本
               │
┌─────────────────────────────────────┐
│         工作区（Draft）             │  ← 可选，用于未保存的修改
│  └─ 关联到一个版本或独立存在         │
└─────────────────────────────────────┘
```

### 5.2 操作流程优化

1. **强制版本流程**：创建片段后必须先创建版本才能编辑弹幕
2. **明确边界**：版本创建后，片段修改需要显式"更新版本"
3. **状态可见性**：UI 始终显示当前操作的数据来源（工作区/哪个版本）

### 5.3 防护机制

1. **操作历史栈**：所有修改操作支持撤销/重做
2. **数据版本标记**：记录数据兼容性版本号
3. **前置条件检查**：所有操作执行前验证前置状态
4. **用户确认**：破坏性操作前要求二次确认
